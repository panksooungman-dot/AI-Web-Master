# User Story

## Purpose

This skill creates well-structured User Stories that describe user needs from the user's perspective and provide sufficient detail for implementation and testing.

User Stories should focus on delivering user value rather than technical implementation.

---

# When to Use

Execute when:

- Planning a sprint
- Refining the product backlog
- Defining new features
- Breaking down epics
- Preparing development work
- Creating acceptance criteria

---

# Objectives

Produce User Stories that are:

- User-centered
- Clear
- Testable
- Independent
- Valuable

---

# Inputs

Expected inputs:

- Product Requirements
- Business Requirements
- Personas
- User Journeys
- Business Rules
- Acceptance Standards

---

# Story Structure

Use the following format:

As a **[user role]**

I want **[goal]**

So that **[business value]**

---

# Story Details

Include:

- Story ID
- Title
- Description
- Business Value
- Priority
- Dependencies
- Assumptions

---

# Acceptance Criteria

Define using Given / When / Then:

- Preconditions
- Trigger
- Expected Result

Example:

Given a registered user

When the user signs in with valid credentials

Then the dashboard is displayed successfully

---

# Definition of Ready

Verify:

- Story understood
- Acceptance criteria defined
- Dependencies identified
- Scope manageable
- Business value clear

---

# Definition of Done

Confirm:

- Code implemented
- Tests passing
- Documentation updated
- Acceptance criteria satisfied
- Product Owner approved

---

# INVEST Validation

Ensure every story is:

- Independent
- Negotiable
- Valuable
- Estimable
- Small
- Testable

---

# Workflow

```text
Review Requirements

↓

Identify User

↓

Define User Goal

↓

Describe Business Value

↓

Write Story

↓

Define Acceptance Criteria

↓

Validate INVEST

↓

Ready for Sprint
```

---

# Outputs

Generate:

- User Story
- Acceptance Criteria
- Ready Checklist
- Done Checklist

---

# Validation Checklist

Before completion verify:

- User identified
- Goal defined
- Business value clear
- Acceptance criteria complete
- INVEST satisfied
- Story ready for implementation

---

# Failure Conditions

Stop if:

- User role is unknown
- Business value is unclear
- Story is too large
- Acceptance criteria are missing
- Requirements are incomplete

---

# Rules

- Write from the user's perspective.
- Focus on value, not implementation.
- Keep stories small.
- Make every story testable.
- Avoid ambiguity.

---

# Success Criteria

This skill succeeds when:

- developers clearly understand the work
- acceptance criteria are measurable
- stories deliver business value
- sprint planning is efficient
